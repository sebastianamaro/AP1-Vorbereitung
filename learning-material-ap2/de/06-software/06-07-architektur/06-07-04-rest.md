# REST

## Lernziele

Nach diesem Kapitel solltest du:
- REST als Architekturstil erklären
- Die HTTP-Methoden und ihre Semantik kennen
- Die wichtigsten REST-Prinzipien nennen

---

## Kerninhalt

**REST** (Representational State Transfer) ist **kein Protokoll, sondern ein Architekturstil** für Web-APIs über HTTP. Eine API, die den Prinzipien folgt, heißt **RESTful**. Datenformat meist **JSON**.

### Prinzipien (Auswahl)

- **Client-Server** – Trennung von UI und Datenhaltung
- **Zustandslos (stateless)** – jeder Request enthält **alle** nötigen Informationen
- **Cacheable** – Antworten können zwischengespeichert werden
- **Uniform Interface** – Ressourcen über **URIs**, einheitliche HTTP-Methoden
- **Layered System**, (optional) Code-on-Demand

### HTTP-Methoden

| Methode | Bedeutung |
|---------|-----------|
| **GET** | Lesen |
| **POST** | Neu anlegen |
| **PUT** | Vollständig ersetzen |
| **PATCH** | Teilweise ändern |
| **DELETE** | Löschen |

```http
GET /api/produkte/42   →   { "id": 42, "name": "Gaming-Maus", "preis": 59.99 }
```

Wichtige **Statuscodes**: 200 OK, 201 Created, 400 Bad Request, 401 Unauthorized, 404 Not Found, 500 Server Error.

---

## Wichtige Begriffe

| Begriff | Erklärung |
|---------|-----------|
| **REST** | Architekturstil für Web-APIs über HTTP |
| **Stateless** | Kein Sitzungszustand auf dem Server |
| **Ressource/URI** | Adressierbares Objekt der API |
| **HTTP-Methoden** | GET/POST/PUT/PATCH/DELETE |

---

## Prüfungstipps

- **REST = Architekturstil, kein Protokoll** – häufige Frage.
- HTTP-Methoden ↔ CRUD (GET=Read, POST=Create, PUT/PATCH=Update, DELETE=Delete).
- **Stateless** erklären können. REST nutzt meist **JSON** (vgl. SOAP = XML).

---

## Übungsaufgabe

**Aufgabe:** Entwirf die REST-Endpunkte (Methode + URI) für Anlegen, Lesen, Ändern und Löschen eines Produkts.
