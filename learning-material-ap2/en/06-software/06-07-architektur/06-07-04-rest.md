# REST

## Learning Objectives

After this chapter you should:
- Explain REST as an architectural style
- Know the HTTP methods and their semantics
- Name the most important REST principles

---

## Core Content

**REST** (Representational State Transfer) is **not a protocol but an architectural style** for web APIs over HTTP. An API that follows the principles is called **RESTful**. The data format is mostly **JSON**.

### Principles (selection)

- **Client-Server** – separation of UI and data storage
- **Stateless** – every request contains **all** necessary information
- **Cacheable** – responses can be cached
- **Uniform Interface** – resources via **URIs**, uniform HTTP methods
- **Layered System**, (optional) Code-on-Demand

### HTTP methods

| Method | Meaning |
|---------|-----------|
| **GET** | Read |
| **POST** | Create new |
| **PUT** | Replace completely |
| **PATCH** | Change partially |
| **DELETE** | Delete |

```http
GET /api/produkte/42   →   { "id": 42, "name": "Gaming-Maus", "preis": 59.99 }
```

Important **status codes**: 200 OK, 201 Created, 400 Bad Request, 401 Unauthorized, 404 Not Found, 500 Server Error.

---

## Key Terms

| Term | Explanation |
|---------|-----------|
| **REST** | Architectural style for web APIs over HTTP |
| **Stateless** | No session state on the server |
| **Resource/URI** | Addressable object of the API |
| **HTTP methods** | GET/POST/PUT/PATCH/DELETE |

---

## Exam Tips

- **REST = architectural style, not a protocol** – a frequent question.
- HTTP methods ↔ CRUD (GET=Read, POST=Create, PUT/PATCH=Update, DELETE=Delete).
- Be able to explain **stateless**. REST mostly uses **JSON** (cf. SOAP = XML).
