# SQL Joins

## Learning Objectives

After this chapter you should be able to:
- Distinguish the four join types
- Link tables via primary/foreign keys
- Choose the appropriate join for a question

---

## Core Content

A **JOIN** links rows from several tables via a condition (usually **PK = FK**).

| Join | Result |
|------|--------|
| **INNER JOIN** | only rows with a **match in both** tables |
| **LEFT (OUTER) JOIN** | all rows on the **left** + matching ones on the right (otherwise NULL) |
| **RIGHT (OUTER) JOIN** | all rows on the **right** + matching ones on the left (otherwise NULL) |
| **FULL OUTER JOIN** | **all** rows from both tables (missing side NULL) |

```sql
-- INNER: nur Kunden mit Bestellungen
SELECT k.name, b.artikel
FROM kunden k
INNER JOIN bestellungen b ON k.kunden_id = b.kunden_id;

-- LEFT: alle Kunden, auch ohne Bestellung
SELECT k.name, b.artikel
FROM kunden k
LEFT JOIN bestellungen b ON k.kunden_id = b.kunden_id;
```

- **INNER** = intersection; **LEFT/RIGHT** = one side complete; **FULL** = union.
- **FULL OUTER JOIN** is not supported by all DBMS (e.g. MySQL → emulate it with `UNION` of LEFT+RIGHT).

---

## Key Terms

| Term | Explanation |
|------|-------------|
| **INNER JOIN** | Only matching rows |
| **LEFT/RIGHT JOIN** | One table complete, the other with NULL |
| **FULL OUTER JOIN** | All rows from both tables |
| **ON condition** | Linking criterion (usually PK = FK) |

---

## Exam Tips

- **INNER vs. LEFT** is the classic: "customers **without** an order too" → LEFT JOIN.
- When writing by hand, **abbreviate table names** (`k`, `b`) and use aliases – it saves time.
- Always specify the **ON condition** (otherwise you get a Cartesian product!).

---

## Practice Exercise

**Task:** Output all customers with their orders – (a) only customers who have ordered, (b) all customers including those without an order. Which join in each case?

<details>
<summary>Solution</summary>

(a) INNER JOIN, (b) LEFT JOIN (kunden on the left).

</details>
