# Análisis de Necesidades

## Objetivos de Aprendizaje

Después de este capítulo, usted debería ser capaz de:
- Explicar el propósito de un análisis de necesidades
- Conocer los pasos de un análisis de necesidades
- Nombrar métodos para la evaluación de necesidades
- Comprender la conexión con las especificaciones de requisitos y funcional

---

## Contenido Principal

### ¿Qué es un Análisis de Necesidades?

Un **análisis de necesidades** determina las necesidades y requisitos reales como base para las decisiones y la planificación del proyecto. Responde a la pregunta: **"¿Qué se necesita realmente?"**

---

### Objetivos del Análisis de Necesidades

- **Comprender necesidades:** ¿Qué necesita realmente el cliente?
- **Aclarar requisitos:** ¿Qué funciones se requieren?
- **Establecer prioridades:** ¿Qué es importante, qué es opcional?
- **Crear fundamento:** Base para la especificación de requisitos y propuesta
- **Evitar decisiones erróneas:** Identificar la solución correcta

---

### Procedimiento para el Análisis de Necesidades

```
+---------------+     +---------------+     +---------------+
|  1. RECOPILACIÓN| -> | 2. ANÁLISIS   | -> | 3. DOCUMENTACIÓN|
|               |     |               |     |               |
| Recopilar     |     | Evaluar y     |     | Registrar     |
| datos         |     | priorizar     |     | resultados    |
|               |     | requisitos    |     |               |
+---------------+     +---------------+     +---------------+
```

---

### Fase 1: Recopilación

**Objetivo:** Recopilar información sobre necesidades

**Métodos:**

| Método | Descripción | Adecuado para |
|--------|-------------|---------------|
| **Entrevista** | Conversaciones individuales con stakeholders | Información profunda |
| **Cuestionario** | Encuesta escrita estructurada | Muchos participantes |
| **Taller** | Trabajo grupal con stakeholders | Soluciones creativas |
| **Observación** | Análisis del estado actual | Comprensión de procesos |
| **Análisis de Documentos** | Evaluación de documentos existentes | Información de antecedentes |

---

### Fase 2: Análisis

**Objetivo:** Evaluar y priorizar requisitos

**Actividades:**
- Categorizar requisitos (funcionales/no funcionales)
- Identificar dependencias
- Establecer prioridades (ej., método MoSCoW)
- Verificar viabilidad
- Resolver contradicciones

**Priorización MoSCoW:**
| Categoría | Significado |
|-----------|-------------|
| **Must have** | Absolutamente requerido |
| **Should have** | Importante pero no crítico |
| **Could have** | Deseable |
| **Won't have** | No en este proyecto |

---

### Fase 3: Documentación

**Objetivo:** Registrar resultados y alinear con stakeholders

**Resultados:**
- Catálogo de requisitos
- Lista priorizada de requisitos
- Especificación de requisitos (base)
- Requisitos alineados con stakeholders

---

### Identificación de Stakeholders

En el análisis de necesidades, todos los **stakeholders** relevantes deben ser involucrados:

| Stakeholder | Interés |
|-------------|---------|
| **Cliente** | Objetivos del proyecto, presupuesto |
| **Usuarios finales** | Usabilidad, funciones |
| **Departamento de TI** | Integración, mantenibilidad |
| **Gerencia** | Viabilidad económica, estrategia |
| **Socios externos** | Interfaces |

---

### Requisitos del Cliente en la Prestación de Servicios

Los **requisitos del cliente** son demandas específicas que un cliente hace a un producto o servicio. Influyen significativamente en:

- Planificación del proyecto
- Ejecución del proyecto
- Control del proyecto

**Importante:** ¡Los requisitos del cliente deben estar claramente documentados y comprendidos!

---

## Términos Clave

| Término | Explicación |
|---------|-------------|
| **Análisis de Necesidades** | Determinación de necesidades y requisitos reales |
| **Stakeholder** | Todas las personas/grupos con interés en el proyecto |
| **Requisito** | Característica o función deseada |
| **Requisito Funcional** | Lo que el sistema debe hacer |
| **Requisito No Funcional** | Cómo debe ser el sistema (calidad) |
| **MoSCoW** | Método de priorización (Must/Should/Could/Won't) |

## Consejos para el Examen

### Preguntas Frecuentes del Examen
- ¿Cuál es el propósito de un análisis de necesidades?
- ¿Qué métodos existen para la evaluación de necesidades?
- ¿Qué son los stakeholders?

### Importante Saber
- El análisis de necesidades viene ANTES de la especificación de requisitos
- Incluir a todos los stakeholders relevantes
- Los requisitos deben ser priorizados

---

## Ejercicios Prácticos

### Ejercicio 1
Nombre tres métodos para la evaluación de necesidades.

**Solución:**
1. Entrevista
2. Cuestionario
3. Taller (o: Observación, Análisis de Documentos)

### Ejercicio 2
¿Qué significa la abreviatura MoSCoW en la priorización?

**Solución:**
- **M**ust have - Absolutamente requerido
- **S**hould have - Importante pero no crítico
- **C**ould have - Deseable
- **W**on't have - No en este proyecto

### Ejercicio 3
¿Por qué es importante incluir a todos los stakeholders en el análisis de necesidades?

**Solución:** Para capturar todas las perspectivas y requisitos. Si se olvidan stakeholders, pueden pasarse por alto requisitos importantes, lo que luego puede llevar a cambios costosos o al fracaso del proyecto.
