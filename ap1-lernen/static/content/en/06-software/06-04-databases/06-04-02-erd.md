# Entity-Relationship Diagram (ERD)

## Learning Objectives

After this chapter, you should be able to:
- Read and understand ERDs
- Identify entities, attributes and relationships
- Interpret cardinalities in ERDs
- Create simple ERDs

---

## Core Content

### What is an ERD?

An **Entity-Relationship Diagram (ERD)** is a graphical model for representing data structures and their relationships.

```
┌─────────────────────────────────────────────────────────────────┐
│                  ENTITY-RELATIONSHIP DIAGRAM                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   Purpose:                                                      │
│   • Data modeling before database creation                     │
│   • Visualization of data structures                           │
│   • Communication between developers and business department   │
│   • Foundation for database design                             │
│                                                                 │
│   Components:                                                   │
│   • Entities (objects/things)                                  │
│   • Attributes (properties)                                    │
│   • Relationships (relations)                                  │
│   • Cardinalities (quantity specifications)                    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### Elements of an ERD

```
┌────────────────────────────────────────────────────────────────┐
│                    ERD ELEMENTS                                │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│   1. ENTITY                                                    │
│      ┌─────────────┐                                          │
│      │   Customer  │  Rectangle with name                     │
│      └─────────────┘  = "thing" about which data is stored    │
│                                                                │
│   2. ATTRIBUTE                                                 │
│         ○                                                      │
│        ╱ ╲                                                     │
│      Name   Ellipse (or list in entity)                       │
│              = property of the entity                          │
│                                                                │
│   3. PRIMARY KEY ATTRIBUTE                                     │
│         ◎         or  ○                                        │
│        ╱ ╲            ╱ ╲                                      │
│      CustomerNo      CustomerNo (underlined)                   │
│              = unique identifier                               │
│                                                                │
│   4. RELATIONSHIP                                              │
│         ◇                                                      │
│        ╱ ╲                                                     │
│       has    Diamond with connecting lines                    │
│              = how entities are connected                     │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### Chen Notation (Classic)

```
┌────────────────────────────────────────────────────────────────┐
│                   CHEN NOTATION                                │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│        ○ Name         ○ Description                            │
│         \            /                                         │
│      ◎ CustomerNo ─┬──┐              ┌──┬─ ◎ ArticleNo        │
│                  │   │              │  │                       │
│              ┌───┴───┴───┐  1   n ┌┴──┴───┐                   │
│              │  Customer │───◇───│ Article │                  │
│              └───────────┘ buys   └────────┘                  │
│                                                                │
│   Symbols:                                                     │
│   ┌─────────┐ = Entity                                        │
│   ○         = Attribute                                        │
│   ◎         = Key attribute (PK)                              │
│   ◇         = Relationship                                     │
│                                                                │
│   Cardinalities:                                               │
│   1  = exactly one                                             │
│   n (or m) = many                                              │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### Modern/Simplified Notation

```
┌────────────────────────────────────────────────────────────────┐
│             SIMPLIFIED ERD NOTATION                            │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│   Common in exams: Attributes directly in the entity          │
│                                                                │
│   ┌─────────────────┐           ┌─────────────────┐          │
│   │     Customer    │           │      Order      │          │
│   ├─────────────────┤           ├─────────────────┤          │
│   │ *CustomerNo     │ 1       n │ *OrderNo        │          │
│   │  Name           │◄──────────│  CustomerNo (FK)│          │
│   │  FirstName      │           │  Date           │          │
│   │  PostalCode     │           │  Total          │          │
│   │  City           │           │                 │          │
│   └─────────────────┘           └─────────────────┘          │
│                                                                │
│   * = Primary Key                                             │
│   (FK) = Foreign Key                                          │
│   1 ◄──── n = One customer has many orders                   │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### Cardinalities (Relationship Types)

```
┌────────────────────────────────────────────────────────────────┐
│                   CARDINALITIES                                │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│   1:1 (One-to-One)                                            │
│   ┌─────────┐    1        1    ┌─────────┐                   │
│   │  Person │◄────────────────►│Passport │                   │
│   └─────────┘                  └─────────┘                   │
│   Each person has exactly one passport                        │
│   Each passport belongs to exactly one person                 │
│                                                                │
│   ──────────────────────────────────────────────────────────  │
│                                                                │
│   1:n (One-to-Many) - MOST COMMON                            │
│   ┌─────────┐    1        n    ┌──────────────┐              │
│   │ Customer│◄─────────────────│    Order     │              │
│   └─────────┘                  └──────────────┘              │
│   A customer has many orders                                  │
│   An order belongs to exactly one customer                    │
│                                                                │
│   ──────────────────────────────────────────────────────────  │
│                                                                │
│   m:n (Many-to-Many)                                          │
│   ┌─────────┐    n        m    ┌─────────┐                   │
│   │ Student │◄────────────────►│ Course  │                   │
│   └─────────┘                  └─────────┘                   │
│   A student takes many courses                                │
│   A course is taken by many students                          │
│   → Is resolved via junction table!                           │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### Min-Max Notation (Crow's Foot)

```
┌────────────────────────────────────────────────────────────────┐
│                CROW'S FOOT NOTATION                            │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│   Symbols at relationship ends:                                │
│                                                                │
│   ───────│       exactly one (1)                               │
│   ───────○       zero or one (0..1)                            │
│   ──────<│       one or more (1..*)                            │
│   ──────<○       zero or more (0..*)                           │
│                                                                │
│   Example:                                                     │
│   ┌─────────┐         ┌──────────────┐                       │
│   │ Customer│──────│──○<──│   Order    │                      │
│   └─────────┘         └──────────────┘                       │
│        │    │                   │                             │
│        │    │                   └── A customer has            │
│        │    │                       0 or more orders          │
│        │    └── An order belongs to exactly 1 customer        │
│        └── Customer                                           │
│                                                                │
│   Reading direction: Read away from the entity!               │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### Complete Example: Online Shop

```
┌────────────────────────────────────────────────────────────────┐
│               EXAMPLE: ONLINE SHOP ERD                         │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│   ┌─────────────────┐                                         │
│   │     Customer    │                                         │
│   ├─────────────────┤                                         │
│   │ *CustomerNo     │                                         │
│   │  Name           │                                         │
│   │  Email          │                                         │
│   │  Address        │                                         │
│   └────────┬────────┘                                         │
│            │ 1                                                 │
│            │                                                   │
│            │ n                                                 │
│   ┌────────┴────────┐                                         │
│   │      Order      │                                         │
│   ├─────────────────┤                                         │
│   │ *OrderNo        │                                         │
│   │  CustomerNo (FK)│                                         │
│   │  OrderDate      │                                         │
│   │  Status         │                                         │
│   └────────┬────────┘                                         │
│            │ 1                                                 │
│            │                                                   │
│            │ n                                                 │
│   ┌────────┴────────┐           ┌─────────────────┐          │
│   │    LineItem     │    n    1 │     Article     │          │
│   ├─────────────────┤◄──────────├─────────────────┤          │
│   │ *LineItemNo     │           │ *ArticleNo      │          │
│   │  OrderNo (FK)   │           │  Description    │          │
│   │  ArticleNo (FK) │           │  Price          │          │
│   │  Quantity       │           │  Stock          │          │
│   │  UnitPrice      │           │                 │          │
│   └─────────────────┘           └─────────────────┘          │
│                                                                │
│   Reading example:                                            │
│   - A customer has n orders (1:n)                            │
│   - An order has n line items (1:n)                          │
│   - A line item refers to 1 article (n:1)                    │
│   - An article can be in n line items (1:n)                  │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### ERD to Tables

```
┌────────────────────────────────────────────────────────────────┐
│               FROM ERD TO DATABASE                             │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│   Conversion rules:                                            │
│                                                                │
│   1. Each entity becomes a table                              │
│   2. Each attribute becomes a column                          │
│   3. The primary key is adopted                               │
│   4. For 1:n: FK in the "n" table                            │
│   5. For m:n: New junction table with two FKs                 │
│                                                                │
│   ERD Example:                                                 │
│   ┌─────────┐  1       n  ┌─────────┐                        │
│   │Department│◄───────────│Employee │                        │
│   └─────────┘             └─────────┘                        │
│                                                                │
│   Resulting tables:                                            │
│   ┌──────────────────┐    ┌────────────────────┐             │
│   │   Department     │    │     Employee       │             │
│   ├──────────────────┤    ├────────────────────┤             │
│   │ DeptNo PK        │◄───│ EmployeeNo PK      │             │
│   │ Description      │    │ Name               │             │
│   └──────────────────┘    │ DeptNo FK ─────────┘             │
│                           └────────────────────┘             │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### Tips for Creating ERDs

| Step | Action |
|------|--------|
| 1 | Identify entities (nouns) |
| 2 | Find attributes for each entity |
| 3 | Define primary keys |
| 4 | Determine relationships between entities |
| 5 | Set cardinalities (1:1, 1:n, m:n) |
| 6 | Resolve m:n relationships into junction tables |

## Exam Tips

### Common Exam Questions

- Read the ERD and explain the relationships
- What cardinality exists between the entities?
- Create an ERD for a given scenario

### Important to Remember

- **Entity** = Rectangle
- **1:n** = One to Many (most common relationship)
- **m:n** = Is resolved via junction table
- **FK always at "n"** (the "many" side)
- ERD shows structure, not data

## Practice Exercises

### Exercise 1
What cardinality exists between "Author" and "Book" (a book has multiple authors, an author writes multiple books)?

**Solution:**
**m:n (Many-to-Many):** An author can write multiple books, and a book can have multiple authors.

This relationship must be resolved via a junction table:
```
Author ◄───── Book_Author ─────► Book
       n:1                 1:n
```

The junction table "Book_Author" contains the foreign keys of both tables.

### Exercise 2
Create a simple ERD for a medical practice with: Patient, Doctor, Appointment

**Solution:**
```
┌─────────────────┐           ┌─────────────────┐
│     Patient     │           │     Doctor      │
├─────────────────┤           ├─────────────────┤
│ *PatientNo      │           │ *DoctorNo       │
│  Name           │           │  Name           │
│  DateOfBirth    │           │  Specialty      │
│  Phone          │           │                 │
└────────┬────────┘           └────────┬────────┘
         │ 1                           │ 1
         │                             │
         │ n                           │ n
         └──────────┬──────────────────┘
                    │
           ┌────────┴────────┐
           │   Appointment   │
           ├─────────────────┤
           │ *AppointmentNo  │
           │  PatientNo (FK) │
           │  DoctorNo (FK)  │
           │  Date           │
           │  Time           │
           └─────────────────┘
```

### Exercise 3
Why must an m:n relationship be resolved via a junction table?

**Solution:**
An m:n relationship cannot be directly represented in a relational database because:

1. **An attribute cannot contain multiple values:** If one wanted to store multiple foreign keys in one table (e.g., all courses of a student), it would violate the rule of atomicity.

2. **Redundancy:** One would either have to create many columns or store data multiple times.

The solution is a **junction table** that contains two foreign keys and thus establishes the link.
