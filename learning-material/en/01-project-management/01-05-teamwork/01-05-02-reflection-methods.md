# Reflection Methods

## Learning Objectives

After this chapter, you should be able to:
- Understand the purpose of reflection methods
- Explain Lessons Learned
- Know the importance of a feedback culture
- Know when reflection takes place in a project

---

## Core Content

### Why Reflection?

Reflection serves to learn from experiences and continuously improve. In project management and agile methods, regular reflection is an important component.

**Goals of Reflection:**
- Learn from mistakes
- Identify and repeat successes
- Improve processes
- Strengthen team cohesion

---

### Lessons Learned

#### Definition

**Lessons Learned** refers to the post-project analysis to systematically learn from experiences and events and make these insights usable for future projects.

#### Typical Questions

| Category | Questions |
|----------|-----------|
| **Successes** | What worked well? What should we keep doing? |
| **Problems** | What didn't work? What were the causes? |
| **Surprises** | What was unexpected? How did we react? |
| **Improvements** | What would we do differently next time? |

#### Process of a Lessons Learned Session

```
+-------------+    +-------------+    +-------------+    +-------------+
| 1. COLLECT  | -> | 2. CLUSTER  | -> | 3. EVALUATE | -> |4. DOCUMENT  |
|             |    |             |    |             |    |             |
| Gather      |    | Group       |    | Prioritize  |    | Record      |
| experiences |    | topics      |    | What is     |    | for the     |
|             |    |             |    | important?  |    | future      |
+-------------+    +-------------+    +-------------+    +-------------+
```

#### Timing
- **Project closure:** Comprehensive analysis
- **After milestones:** Interim review
- **With Scrum:** Sprint Retrospective

### Feedback Culture

#### Definition

A **feedback culture** means that feedback is given regularly and openly. It is the foundation for continuous improvement.

#### Characteristics of Good Feedback

| Characteristic | Description |
|----------------|-------------|
| **Honest** | Truthful, not sugarcoated |
| **Respectful** | Appreciative, not hurtful |
| **Constructive** | With suggestions for improvement |
| **Timely** | As soon as possible after the event |
| **Specific** | Related to specific situations |
| **Helpful** | Useful for the recipient |

#### Feedback Rules

**Giving Feedback:**
- Use I-statements ("I observed...")
- Describe behavior, don't judge person
- Give concrete examples
- Make suggestions for improvement

**Receiving Feedback:**
- Listen without interrupting
- Ask questions if unclear
- Don't justify yourself
- Be grateful for the feedback

### Reflection in Scrum: Sprint Retrospective

The **Sprint Retrospective** is the Scrum meeting for team reflection:

| Aspect | Details |
|--------|---------|
| **When** | At the end of each Sprint, after Sprint Review |
| **Duration** | Maximum 3 hours (for 4-week Sprint) |
| **Participants** | Scrum Team (PO, SM, Development Team) |
| **Focus** | The process, not the product |

**Typical Questions:**
1. What went well in the Sprint?
2. What can we improve?
3. What concrete actions will we take?

---

### Methods for Reflection

#### Start-Stop-Continue

| Category | Question |
|----------|----------|
| **Start** | What should we start doing? |
| **Stop** | What should we stop doing? |
| **Continue** | What should we continue doing? |

#### 4-L Retrospective

| L | Meaning |
|---|---------|
| **Liked** | What did I like? |
| **Learned** | What did I learn? |
| **Lacked** | What was I missing? |
| **Longed for** | What did I wish for? |

## Key Terms

| Term | Explanation |
|------|-------------|
| **Lessons Learned** | Systematic analysis of project experiences |
| **Feedback Culture** | Environment where open feedback is given |
| **Sprint Retrospective** | Scrum meeting for process improvement |
| **Constructive Feedback** | Feedback with suggestions for improvement |
| **Continuous Improvement** | Constant striving for optimization |

## Exam Tips

### Frequent Exam Questions
- What is meant by Lessons Learned?
- When does the Sprint Retrospective take place?
- What is the difference between Sprint Review and Retrospective?

### Remember
- **Sprint Review** = Product feedback (What did we build?)
- **Sprint Retrospective** = Process feedback (How did we work?)

---

## Practice Exercises

### Exercise 1
What is meant by Lessons Learned?

**Solution:** The post-project analysis to systematically learn from experiences and make these insights usable for future projects.

### Exercise 2
Name three characteristics of good feedback.

**Solution:**
1. Honest
2. Respectful
3. Constructive (or: timely, specific, helpful)

### Exercise 3
What is the difference between Sprint Review and Sprint Retrospective?

**Solution:**
- **Sprint Review:** Presentation of work results, feedback on the PRODUCT
- **Sprint Retrospective:** Analysis of collaboration, improvement of the PROCESS
