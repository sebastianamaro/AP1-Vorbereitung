# Entity-Relationship-Diagramm (ERD)

## Lernziele

Nach diesem Kapitel solltest du:
- ERDs lesen und verstehen können
- Entitäten, Attribute und Beziehungen identifizieren können
- Kardinalitäten in ERDs interpretieren können
- Einfache ERDs erstellen können

---

## Kerninhalt

### Was ist ein ERD?

Ein **Entity-Relationship-Diagramm (ERD)** ist ein grafisches Modell zur Darstellung von Datenstrukturen und deren Beziehungen.

```
┌─────────────────────────────────────────────────────────────────┐
│                  ENTITY-RELATIONSHIP-DIAGRAMM                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   Zweck:                                                       │
│   • Datenmodellierung vor der Datenbank-Erstellung            │
│   • Visualisierung von Datenstrukturen                         │
│   • Kommunikation zwischen Entwicklern und Fachbereich        │
│   • Grundlage für Datenbank-Design                             │
│                                                                 │
│   Bestandteile:                                                │
│   • Entitäten (Objekte/Dinge)                                 │
│   • Attribute (Eigenschaften)                                  │
│   • Beziehungen (Relationen)                                   │
│   • Kardinalitäten (Mengenangaben)                            │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### Elemente eines ERD

```
┌────────────────────────────────────────────────────────────────┐
│                    ERD-ELEMENTE                                │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│   1. ENTITÄT (Entity)                                         │
│      ┌─────────────┐                                          │
│      │   Kunde     │  Rechteck mit Namen                     │
│      └─────────────┘  = "Ding", über das Daten gespeichert   │
│                         werden                                │
│                                                                │
│   2. ATTRIBUT                                                  │
│         ○                                                      │
│        ╱ ╲                                                     │
│      Name   Ellipse (oder Liste in der Entität)              │
│              = Eigenschaft der Entität                        │
│                                                                │
│   3. PRIMÄRSCHLÜSSEL-ATTRIBUT                                 │
│         ◎         oder  ○                                      │
│        ╱ ╲              ╱ ╲                                    │
│      KundenNr         KundenNr (unterstrichen)               │
│              = eindeutiges Identifikationsmerkmal             │
│                                                                │
│   4. BEZIEHUNG (Relationship)                                 │
│         ◇                                                      │
│        ╱ ╲                                                     │
│       hat    Raute mit Verbindungslinien                     │
│              = Wie Entitäten zusammenhängen                  │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### Chen-Notation (klassisch)

```
┌────────────────────────────────────────────────────────────────┐
│                   CHEN-NOTATION                                │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│        ○ Name         ○ Bezeichnung                           │
│         \            /                                         │
│      ◎ KundenNr ─┬──┐              ┌──┬─ ◎ ArtikelNr         │
│                  │   │              │  │                       │
│              ┌───┴───┴───┐  1   n ┌┴──┴───┐                   │
│              │   Kunde   │───◇───│ Artikel │                  │
│              └───────────┘ kauft  └────────┘                  │
│                                                                │
│   Symbole:                                                    │
│   ┌─────────┐ = Entität                                      │
│   ○         = Attribut                                        │
│   ◎         = Schlüsselattribut (PK)                         │
│   ◇         = Beziehung                                       │
│                                                                │
│   Kardinalitäten:                                             │
│   1  = genau eins                                             │
│   n (oder m) = viele                                          │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### Moderne/Vereinfachte Notation

```
┌────────────────────────────────────────────────────────────────┐
│             VEREINFACHTE ERD-NOTATION                          │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│   Häufig in Prüfungen: Attribute direkt in der Entität       │
│                                                                │
│   ┌─────────────────┐           ┌─────────────────┐          │
│   │     Kunde       │           │    Bestellung   │          │
│   ├─────────────────┤           ├─────────────────┤          │
│   │ *KundenNr       │ 1       n │ *BestellNr      │          │
│   │  Name           │◄──────────│  KundenNr (FK)  │          │
│   │  Vorname        │           │  Datum          │          │
│   │  PLZ            │           │  Summe          │          │
│   │  Ort            │           │                 │          │
│   └─────────────────┘           └─────────────────┘          │
│                                                                │
│   * = Primärschlüssel                                        │
│   (FK) = Fremdschlüssel                                      │
│   1 ◄──── n = Ein Kunde hat viele Bestellungen               │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

---

### Kardinalitäten (Beziehungstypen)

```
┌────────────────────────────────────────────────────────────────┐
│                   KARDINALITÄTEN                               │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│   1:1 (Eins-zu-Eins)                                         │
│   ┌─────────┐    1        1    ┌─────────┐                   │
│   │  Person │◄────────────────►│  Pass   │                   │
│   └─────────┘                  └─────────┘                   │
│   Jede Person hat genau einen Pass                           │
│   Jeder Pass gehört zu genau einer Person                    │
│                                                                │
│   ──────────────────────────────────────────────────────────  │
│                                                                │
│   1:n (Eins-zu-Viele) - AM HÄUFIGSTEN                        │
│   ┌─────────┐    1        n    ┌──────────────┐              │
│   │ Kunde   │◄─────────────────│  Bestellung  │              │
│   └─────────┘                  └──────────────┘              │
│   Ein Kunde hat viele Bestellungen                           │
│   Eine Bestellung gehört zu genau einem Kunden               │
│                                                                │
│   ──────────────────────────────────────────────────────────  │
│                                                                │
│   m:n (Viele-zu-Viele)                                       │
│   ┌─────────┐    n        m    ┌─────────┐                   │
│   │ Student │◄────────────────►│  Kurs   │                   │
│   └─────────┘                  └─────────┘                   │
│   Ein Student besucht viele Kurse                            │
│   Ein Kurs wird von vielen Studenten besucht                 │
│   → Wird über Zwischentabelle aufgelöst!                     │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### Min-Max-Notation (Crow's Foot)

```
┌────────────────────────────────────────────────────────────────┐
│                CROW'S FOOT NOTATION                            │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│   Symbole am Beziehungsende:                                  │
│                                                                │
│   ───────│       genau eins (1)                               │
│   ───────○       null oder eins (0..1)                        │
│   ──────<│       eins oder mehr (1..*)                        │
│   ──────<○       null oder mehr (0..*)                        │
│                                                                │
│   Beispiel:                                                   │
│   ┌─────────┐         ┌──────────────┐                       │
│   │ Kunde   │──────│──○<──│ Bestellung │                      │
│   └─────────┘         └──────────────┘                       │
│        │    │                   │                             │
│        │    │                   └── Ein Kunde hat             │
│        │    │                       0 oder mehr Bestellungen │
│        │    └── Eine Bestellung gehört zu genau 1 Kunden     │
│        └── Kunde                                              │
│                                                                │
│   Leserichtung: Von der Entität weg lesen!                   │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### Vollständiges Beispiel: Online-Shop

```
┌────────────────────────────────────────────────────────────────┐
│               BEISPIEL: ONLINE-SHOP ERD                        │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│   ┌─────────────────┐                                         │
│   │     Kunde       │                                         │
│   ├─────────────────┤                                         │
│   │ *KundenNr       │                                         │
│   │  Name           │                                         │
│   │  Email          │                                         │
│   │  Adresse        │                                         │
│   └────────┬────────┘                                         │
│            │ 1                                                 │
│            │                                                   │
│            │ n                                                 │
│   ┌────────┴────────┐                                         │
│   │   Bestellung    │                                         │
│   ├─────────────────┤                                         │
│   │ *BestellNr      │                                         │
│   │  KundenNr (FK)  │                                         │
│   │  Bestelldatum   │                                         │
│   │  Status         │                                         │
│   └────────┬────────┘                                         │
│            │ 1                                                 │
│            │                                                   │
│            │ n                                                 │
│   ┌────────┴────────┐           ┌─────────────────┐          │
│   │    Position     │    n    1 │     Artikel     │          │
│   ├─────────────────┤◄──────────├─────────────────┤          │
│   │ *PositionsNr    │           │ *ArtikelNr      │          │
│   │  BestellNr (FK) │           │  Bezeichnung    │          │
│   │  ArtikelNr (FK) │           │  Preis          │          │
│   │  Menge          │           │  Bestand        │          │
│   │  Einzelpreis    │           │                 │          │
│   └─────────────────┘           └─────────────────┘          │
│                                                                │
│   Lesebeispiel:                                              │
│   - Ein Kunde hat n Bestellungen (1:n)                       │
│   - Eine Bestellung hat n Positionen (1:n)                   │
│   - Eine Position verweist auf 1 Artikel (n:1)               │
│   - Ein Artikel kann in n Positionen sein (1:n)              │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### ERD zu Tabellen

```
┌────────────────────────────────────────────────────────────────┐
│               VOM ERD ZUR DATENBANK                            │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│   Umwandlungsregeln:                                          │
│                                                                │
│   1. Jede Entität wird zu einer Tabelle                      │
│   2. Jedes Attribut wird zu einer Spalte                     │
│   3. Der Primärschlüssel wird übernommen                     │
│   4. Bei 1:n: FK in die "n"-Tabelle                          │
│   5. Bei m:n: Neue Zwischentabelle mit zwei FKs              │
│                                                                │
│   Beispiel ERD:                                               │
│   ┌─────────┐  1       n  ┌─────────┐                        │
│   │ Abteilung│◄────────────│Mitarbeiter│                      │
│   └─────────┘             └─────────┘                        │
│                                                                │
│   Ergebnis-Tabellen:                                          │
│   ┌──────────────────┐    ┌────────────────────┐             │
│   │    Abteilung     │    │    Mitarbeiter     │             │
│   ├──────────────────┤    ├────────────────────┤             │
│   │ AbtNr PK         │◄───│ MitarbNr PK        │             │
│   │ Bezeichnung      │    │ Name               │             │
│   └──────────────────┘    │ AbtNr FK ──────────┘             │
│                           └────────────────────┘             │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### Tipps zur ERD-Erstellung

| Schritt | Aktion |
|---------|--------|
| 1 | Entitäten identifizieren (Substantive) |
| 2 | Attribute zu jeder Entität finden |
| 3 | Primärschlüssel festlegen |
| 4 | Beziehungen zwischen Entitäten bestimmen |
| 5 | Kardinalitäten festlegen (1:1, 1:n, m:n) |
| 6 | m:n-Beziehungen in Zwischentabellen auflösen |

## Wichtige Begriffe

| Begriff | Erklärung |
|---------|-----------|
| **ERD** | Entity-Relationship-Diagramm |
| **Entität** | Ein Ding/Objekt, über das Daten gespeichert werden |
| **Attribut** | Eigenschaft einer Entität |
| **Beziehung** | Verbindung zwischen Entitäten |
| **Kardinalität** | Anzahlverhältnis (1:1, 1:n, m:n) |
| **Chen-Notation** | Klassische ERD-Darstellung mit Rauten |
| **Crow's Foot** | Notation mit Krähenfuß-Symbolen |

## Prüfungstipps

### Häufige Prüfungsfragen

- Lesen Sie das ERD und erklären Sie die Beziehungen
- Welche Kardinalität besteht zwischen den Entitäten?
- Erstellen Sie ein ERD für ein gegebenes Szenario

### Wichtig zu merken

- **Entität** = Rechteck
- **1:n** = Ein zu Viele (häufigste Beziehung)
- **m:n** = Wird über Zwischentabelle aufgelöst
- **FK immer bei "n"** (der "Viele"-Seite)
- ERD zeigt Struktur, nicht Daten

## Übungsaufgaben

### Aufgabe 1
Welche Kardinalität besteht zwischen "Autor" und "Buch" (ein Buch hat mehrere Autoren, ein Autor schreibt mehrere Bücher)?

**Lösung:**
**m:n (Viele-zu-Viele):** Ein Autor kann mehrere Bücher schreiben, und ein Buch kann mehrere Autoren haben.

Diese Beziehung muss über eine Zwischentabelle aufgelöst werden:
```
Autor ◄───── Buch_Autor ─────► Buch
       n:1               1:n
```

Die Zwischentabelle "Buch_Autor" enthält die Fremdschlüssel beider Tabellen.

### Aufgabe 2
Erstellen Sie ein einfaches ERD für eine Arztpraxis mit: Patient, Arzt, Termin

**Lösung:**
```
┌─────────────────┐           ┌─────────────────┐
│     Patient     │           │      Arzt       │
├─────────────────┤           ├─────────────────┤
│ *PatientNr      │           │ *ArztNr         │
│  Name           │           │  Name           │
│  Geburtsdatum   │           │  Fachgebiet     │
│  Telefon        │           │                 │
└────────┬────────┘           └────────┬────────┘
         │ 1                           │ 1
         │                             │
         │ n                           │ n
         └──────────┬──────────────────┘
                    │
           ┌────────┴────────┐
           │     Termin      │
           ├─────────────────┤
           │ *TerminNr       │
           │  PatientNr (FK) │
           │  ArztNr (FK)    │
           │  Datum          │
           │  Uhrzeit        │
           └─────────────────┘
```

### Aufgabe 3
Warum muss eine m:n-Beziehung über eine Zwischentabelle aufgelöst werden?

**Lösung:**
Eine m:n-Beziehung kann nicht direkt in einer relationalen Datenbank abgebildet werden, weil:

1. **Ein Attribut kann nicht mehrere Werte enthalten:** Würde man in einer Tabelle mehrere Fremdschlüssel speichern wollen (z.B. alle Kurse eines Studenten), würde das die Regel der Atomarität verletzen.

2. **Redundanz:** Man müsste entweder viele Spalten anlegen oder Daten mehrfach speichern.

Die Lösung ist eine **Zwischentabelle**, die zwei Fremdschlüssel enthält und so die Verknüpfung herstellt.
